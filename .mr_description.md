## Why

`initFromBackend().then(...)` had no `.catch`. Inside it, `Promise.all([listRepos(), listTickets()])` coupled the two fetches: if the ticket provider errored (bad Linear/Jira credentials, provider down) or the daemon wasn't up yet, the whole init rejected — `repos` was never set, and `MissionControl.svelte:165` (`showOnboarding = $repos.length === 0`) showed **"Add a repository to get started"** to a user who has repos and running sessions. No toast, no retry — the failure was invisible and the message actively wrong.

## What changed

- **stores.ts**: `initFromBackend` now fetches repos, tickets, and sessions independently with per-call `try/catch` + error toasts. Added `initialLoadLoading` and `initialLoadError` stores to track the initial load state. Added `retryInitialLoad()` function.

- **MissionControl.svelte**: Now consumes the loading/error stores:
  - Shows loading spinner while initial fetch is in progress
  - Shows retry UI (with error message and "Try again" button) when load fails
  - Only shows first-run onboarding when NOT loading, NO error, AND genuinely no repos
  - Added styling for loading/error states

- **App.svelte**: No changes needed — the existing `initFromBackend().then(...)` chain works because the function now never rejects (all errors handled internally).
